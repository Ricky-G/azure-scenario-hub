● The core question — true mTLS validation at APIM

Does App Gateway forward the client certificate to APIM, and does APIM validate it against Key Vault?

Yes. APIM reconstructs the forwarded certificate and validates it under the selected model before authorizing per client - a pinned-thumbprint allow list, or a real cryptographic chain-of-trust signature check. A certificate from an untrusted issuer is rejected even though the gateway forwarded it unvalidated: APIM is the trust anchor, and Key Vault holds the trust material.

Two validation models · both fully supported

There are two legitimate ways to decide whether a forwarded certificate is trustworthy, and this scenario ships both. Choose one with the certValidationMode parameter (pinned or chain). Neither is universally “correct” — they trade explicit control against operational overhead. Both were run live; see the evidence below.

🔑 Model A · Pinned thumbprint

Accept only certificates on a Key Vault allow list.

Trust test
SHA-1 thumbprint is on the allow list (and issuer matches the Root CA).
New client
Add its thumbprint to Key Vault first.
Strength
Most restrictive; an over-permissive CA can’t widen access.
Cost
Per-certificate maintenance.
Best for
A small, known set of clients.
client3 (CA-signed, not listed) → rejected · spoofed → rejected (not on list)

🔗 Model B · Chain of trust

Accept anything the Root CA cryptographically signed.

Trust test
Root CA public key verifies the certificate’s RSA signature (real VerifyData).
New client
Works automatically once your CA issues it.
Strength
No allow list to maintain; trust the CA, not each cert.
Cost
You must fully trust (and protect) the CA.
Best for
A private CA issuing many / rotating client certs.
client3 (CA-signed, not listed) → accepted · spoofed → rejected (NOT_CA_SIGNED)

“Isn’t this just a thumbprint or name compare?” — No.

Chain mode performs a genuine RSA signature verification with the Key Vault Root CA’s public key. The spoofed certificate proves it: its issuer Distinguished Name is byte-for-byte identical to the real Root CA (CN=mTLS POC Root CA), yet it is rejected with NOT_CA_SIGNED and chainOk=false — because the Root CA’s key never signed it. A string comparison would have been fooled; RSA.VerifyData is not.

Sandbox note: the APIM policy validator blocks both X509Chain and System.Func (both confirmed at deploy time), so chain mode parses the leaf DER inline to recover the TBSCertificate bytes and signature, then verifies with GetRSAPublicKey().VerifyData(...).

How we validated the forwarded certificate in the APIM sandbox

Parse the header, then decide trust in APIM — never at the gateway.

Behind a Layer-7 gateway, APIM's context.Request.Certificate is not the client cert (the gateway opens a separate TLS session to APIM). The client certificate is forwarded as a percent-encoded PEM in X-Client-Cert, set from the gateway's mutual-authentication server variables.

The APIM policy URL-decodes the header, strips the PEM armor, base64-decodes to DER, and builds an X509Certificate2. It then checks the validity window and applies the active model: a Key Vault-sourced thumbprint allow list (pinned) or a real Root CA signature verification (chain).

Sandbox limitations found at deploy time: the APIM policy-expression validator rejects DateTime.ToUniversalTime(), X509Chain, and System.Func. Validity uses DateTime.Now against the certificate’s local NotBefore/NotAfter; chain mode verifies trust by parsing the leaf DER inline (no X509Chain, no lambdas) and calling GetRSAPublicKey().VerifyData(...) with the Key Vault Root CA.

Security verdict

● A strong, defensible pattern

  • Possession — enforced by TLS; a public certificate without its private key is useless.
  • Trust — enforced in APIM under whichever model you pick: a pinned Key Vault allow list, or a real cryptographic chain-of-trust signature check. APIM is the trust anchor either way.
  • Real cryptography, not string matching — chain mode rejects the spoofed certificate (identical issuer name, forged signature) with chainOk=false. Trust is proven by the CA’s key, not by a name.
  • Authorization — per-client path binding in APIM operation policies.
  • Anti-spoofing — the gateway overwrites every X-Client-Cert* header from the real TLS connection, and APIM is reachable only from the gateway (internal VNet), so a forged header can’t be injected or bypass the gateway.

One thing to keep in mind: the gateway itself performs no certificate validation in passthrough, so treat its client_certificate_verification value as informational only — the header overwrite and the internal-VNet lockdown are what make the forwarded certificate trustworthy for APIM to validate.
