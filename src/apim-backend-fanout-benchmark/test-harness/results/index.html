APIM Backend Fan-out Benchmark — Results

Shared Backend + rewrite-uri  vs  one Backend per API · Australia East · APIM Premium 1 unit · .NET 10 backend on Flex Consumption FC1

✅ No measurable difference between the two patterns

Across 1.67 million requests over ~32 minutes of stepped load, APIM-A (shared backend + rewrite-uri) and APIM-B (one backend per API) produced equivalent latency and throughput. All three pass/fail thresholds satisfied. Both patterns are safe to use — choose based on operational simplicity, not performance.

Recommendation

Either pattern is fine for performance. Decide on operational grounds:

  • Shared backend + rewrite-uri — one backend entity to manage, but every API depends on the same backend definition. Cheap policy-eval cost (single string concat per request).
  • One backend per API — clearer 1-to-1 mapping, easier to apply per-backend circuit breakers / health probes / authentication, but more entities to keep in sync.
