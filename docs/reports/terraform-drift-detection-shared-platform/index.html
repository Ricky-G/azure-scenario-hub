Where exactly does Terraform call it “drift”?

A platform team owns shared landing-zone resources in Terraform. An app team adds their own Foundry projects, Cosmos databases and blob containers out-of-band with Bicep. We deployed it for real and measured the precise line where the platform team’s pipeline starts shouting “drift.”

In plain English 🗣️

New to drift? Read this first — it clears up the most common mix-up.

✓
Terraform keeps a checklist of only the things it personally created. Here that's three accounts: the storage account, the Cosmos account, and the Foundry account.
✓
When the app team adds new things inside those accounts — a blob container, a Cosmos database, a Foundry project — Terraform never sees them, because they were never on its checklist. So it says “nothing changed.” This is genuine blindness, not suppression — we did not hide anything.
!
The only time Terraform reports drift is when someone edits a setting on one of those three accounts it created — like adding a tag to the storage account. Now reality no longer matches its notes, so it flags it.
~
ignore_changes is an optional mute button for that tag case only. It tells Terraform “I'm fine with people editing the tags, stop reporting it.” It has nothing to do with the container, database, or project — those were already invisible.

● Invisible to Terraform

App team created brand-new child resources. Platform terraform plan stayed silent — nothing was hidden, Terraform simply never tracked them.
  • blob container appteam-data → in the platform storage account
  • cosmos sql database + container appteam-db / orders
  • foundry project appteam-project
The Line

● Triggers drift

Anything that changes an attribute Terraform already manages on a resource it owns.
  • tag added to the storage account tags is a managed attribute → plan wants to remove it
  • SKU / TLS / network / public-access flags any first-class managed property
  • policy flipping disableLocalAuth policy-as-code is just another out-of-band actor
Bottom line: new, independent ARM child resources are invisible to the platform state. Drift only fires when an out-of-band change touches a property Terraform records for a resource it manages — and ignore_changes only mutes that second case.

Bonus finding — policy-as-code is an out-of-band actor too

An Azure Policy in this subscription created drift before the app team did anything.

The subscription enforces keyless auth. Right after apply, a policy flipped disableLocalAuth → true on the Foundry account and local_authentication_disabled → true on Cosmos. Because those are Terraform-managed attributes, the very first plan reported drift — exactly the same rule as the app-team tag. We aligned the config to the policy (the secure default) to get a clean baseline. Lesson: a Modify/DeployIfNotExists policy lands on the drift side of the line, just like a human with the CLI.

So… where is the line?

NO DRIFT  →  create a new, independent ARM resource under a managed parent
          (blob container · cosmos database · foundry project)

DRIFT      →  modify an attribute Terraform records on a resource it manages
          (a tag · a SKU · a TLS setting · a network rule · a policy flip)

What this means for a platform / app-team split:

  • Hand app teams child resources. Databases, containers and Foundry projects never touch the platform state — no drift, no pipeline noise.
  • Keep account-level properties (SKU, network, TLS, auth mode) as the platform team’s domain — drift there is exactly what you want the pipeline to catch.
  • Use ignore_changes to deliberately cede shared fields (usually tags) without generating false alarms.
  • Remember policy-as-code. Modify/DeployIfNotExists effects change managed attributes and will show up as drift — align your config with the secure baseline.
